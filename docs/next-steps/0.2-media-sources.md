# 0.2 Media Sources — implementation plan

This is the living plan for milestone **0.2 Media Sources** in [`../../ROADMAP.md`](../../ROADMAP.md). Edit as scope shifts; don't leave stale entries.

The goal of 0.2 is to **turn Aether from a shell into a player**: real Plex and real Synology libraries, unified at the `MediaItem` layer, with artwork that actually loads and resume points that survive a launch.

When 0.2 ships, a fresh contributor should be able to:

1. Launch Aether.
2. Sign in to a real Plex account (PIN flow).
3. Pick a real Plex server.
4. Sign in to a real Synology NAS.
5. Browse both libraries from the same Home.
6. Play any title; resume points persist across launches.

---

## Implementation order

1. **Plex skeleton** — networking foundation, PIN auth, DTOs. ✅ *Shipped*
2. **Plex sign-in UI** — display the code + open `plex.tv/link`; observe `PlexAuthClient` poll loop in a `@MainActor` view model. ✅ *Shipped*
3. **Plex server discovery** — fetch `/api/v2/resources`, pick a connection by static ranking (local > non-relay > HTTPS > owned). RTT probing deferred. ✅ *Shipped*
4. **Plex libraries + items** — `/library/sections` and `/library/sections/{key}/all`; map metadata to `MediaItem`; tokenised artwork URLs; `AppSession` swap to `plexSource`. ✅ *Shipped*
5. **Plex stream URLs (direct play)** — first Part's `key`, tokenised. Movies + episodes play in AVPlayer for supported codecs. ✅ *Shipped*
6. **Connection failover + drop the mock** — persist all of a server's connections (ranked); probe `/identity` at runtime so Plex works on the LAN *and* off it (remote / relay). Removed the mock library from the running app. ✅ *Shipped*
7. **Plex transcode fallback** — `/video/:/transcode/universal/start.m3u8` HLS for containers AVPlayer can't direct-play (MKV, AVI, TS…). Per-item decision from `Media.container`: friendly (mp4/m4v/mov) → direct file, else → transcode (`directStream=1` remuxes the common case). ✅ *Shipped*
8. **Show → seasons → episodes** drill-down so TV libraries are browsable past the show poster. ✅ *Shipped*
9. **Synology auth** — DSM session auth or Video Station auth (spike first to decide which).
10. **Synology libraries + items** — shares / collections / items.
11. **Synology stream URLs** — direct play when codec supported; transcode fallback when the server can.
12. **Persistent ResumeStore** — SwiftData-backed; the outbox plumbing for offline writes lands properly in 0.3.
13. **Artwork pipeline** — disk-backed LRU + downsampling inside `CachedAsyncImage` (currently a thin wrapper).

## Plex skeleton — what this PR lands

Concretely, the layer everything else builds on:

- **`AetherCore/Networking/APIClient`** — small protocol around `URLSession.data(for:)`. `URLSessionAPIClient` ships as the production implementation; tests use a recording stub.
- **`AetherCore/Storage/KeychainStore`** — actor wrapper around `kSecClassGenericPassword`. Stores Plex auth tokens, the per-install client identifier, future Synology credentials.
- **`AetherCore/MediaSources/Plex/PlexConfiguration`** — the `X-Plex-*` headers Plex requires on every request, including a stable per-install `clientIdentifier`.
- **`AetherCore/MediaSources/Plex/PlexAPI`** — `Decodable` DTOs for the endpoints we'll call (`PIN`, `Resource`, `Resource.Connection`).
- **`AetherCore/MediaSources/Plex/PlexAuthClient`** — actor implementing the PIN flow: `requestPIN()` → display `code` → `pollForToken(pinID:interval:timeout:)`.
- **`AetherCore/MediaSources/Plex/PlexMediaSource`** — keeps its `MediaSource` shape, now takes `baseURL`, `accessToken`, `configuration`, and an `APIClient`. `libraries()` and `items(in:)` are still stubs — they land in the next PR.

Tests cover: header completeness, PIN JSON decoding (un-authed + authed), Resource JSON decoding, the polling loop reaching success / expired states, and the `PlexMediaSource.request(forPath:)` header set.

## What this PR is not

- **No UI.** Sign-in lives in a follow-up so this PR is reviewable in one sitting.
- **No real server calls in tests.** Only the recording `APIClient` is exercised; integration testing against a real Plex server happens manually.
- **No SwiftData persistence yet.** Keychain stores tokens; `ResumeStore` is still in-memory.
- **No connection picker.** The next PR picks the best connection from `Resource.connections` by RTT.

## Discovery flow

```
PlexSignInView completes
        │  (token persisted to Keychain)
        ▼
AppSession.completePlexSignIn(token:)
        │
        ▼
AppSession.discoverPlexServers()
        │   fetches /api/v2/resources (PlexResourceClient)
        ▼
PlexServerSelector.selectBest(from: resources)
        │   filter: providesServer + has accessToken + has connections
        │   rank:   local +1000, !relay +100, https +10, owned +1
        ▼
PlexServerStore.write(record)     ← Keychain JSON blob
        │
        ▼
AppSession.plexSource = PlexMediaSource(...)
        │
        ▼
DiscoveryState.completed(serverName)
```

On next launch:

```
AppSession.start()
        │
        ▼
restorePlexServer()  ← reads PlexServerStore
        │
        ├── record present  → AppSession.plexSource ready, state = .completed
        └── nothing on file → state = .idle, user can re-run discovery
```

## Ranking rules

Pure, deterministic, no I/O. Each `(server, connection)` pair gets a single integer score; higher wins.

| Bit | Weight | Why |
|-----|--------|-----|
| local connection | +1000 | LAN beats anything WAN — latency, bandwidth, no Plex-relay cost |
| direct (non-relay) | +100 | Direct is reliable; relay has Plex's bandwidth caps |
| HTTPS | +10 | Trust the cert + `plex.direct` hostnames; consistent with Plex's defaults |
| owned server | +1 | Tiebreaker; the user's own server beats a friend-shared one |

### Deliberately out of scope (for now)

- **RTT probing.** A real probe would HEAD `/identity` on each candidate with a short timeout and race the winners. It's a layer that drops in on top of the static score as a tiebreaker; not necessary for the common case (local + direct + https).
- **Friend-server picker UI.** Shared servers are included (they just sit a tier below owned ones). A future Settings flow will let the user explicitly switch among visible servers.
- **Transcoding awareness.** That's a runtime concern Plex handles at playback time; discovery doesn't need to predict it.

## Server selection behavior

- One server selected at a time; replaces any previous selection in Keychain.
- Selection survives launches — `AppSession.start()` reads `PlexServerStore` and rebuilds the `PlexMediaSource`.
- `signOutOfPlex()` clears both the auth token and the server record.
- `discoverPlexServers()` is safe to re-run; it always re-fetches and re-picks. UI exposes this as `Try again`.

## Open questions

- **Client identifier persistence.** The first time Aether launches we'll generate a UUID and store it in Keychain; the next launch reads it. Where does the generation happen? Probably `AppSession.start()` before either source is constructed.
- **Synology auth path.** DSM session API vs Video Station — spike first, decide which gives cleaner stream URLs. (Not blocking the Plex PR.)
- **Per-server vs per-account tokens.** Plex returns a per-server `accessToken` in the resources response — using that scopes blast radius if any one server is compromised. We'll prefer per-server tokens.
