# 0.3 Offline — implementation plan

This is the living plan for milestone **0.3 Offline** in [`../../ROADMAP.md`](../../ROADMAP.md). Edit as scope shifts; don't leave stale entries.

The goal of 0.3 is **travel mode**: a downloaded title plays end-to-end with the network completely off, and the resume position written offline syncs back the moment a source is reachable again.

When 0.3 ships, a fresh user should be able to:

1. Browse a Plex (or Synology) library and tap "Download" on a title.
2. Close the app, walk out the door, get on a plane (or just toggle airplane mode).
3. Open Aether, find the title in **Downloads**, play it, scrub it, pause it — all offline.
4. Reconnect and see the resume point arrive on the server within seconds.

---

## Implementation order

The order matters — later items depend on earlier ones being solid.

1. **`DownloadManager` actor** — owns the active downloads, the background `URLSession` delegate, and the per-task progress fan-out. Lives in `AetherCore/Downloads/`.
2. **`DownloadStore` (SwiftData)** — persists download metadata: item id, source, local URL, byte size, dates, state. Survives launches and OS-killed downloads.
3. **`DiskBudget` actor** — knows the user's configured cap (default 8 GB), the current footprint, and which titles are eligible for eviction. Pure policy; no I/O of its own.
4. **Download UI on Detail** — a `Download` row next to Play that flips through `Not downloaded → Queued → Downloading <%> → Downloaded → Removing`. `AetherButton` variants for the action.
5. **Downloads tab on Home (or a new tab/section)** — calm list of downloaded titles with progress, size, and a swipe / context-menu **Remove**. The same surface shows queued + in-flight downloads.
6. **Offline playback path** — `MediaItem.streamURL` resolution prefers a local URL when one exists in `DownloadStore`. Remote fallback is opt-in (the user explicitly chose to play remote from a downloaded surface), not silent.
7. **Resume outbox** — resume writes that happen offline land in a small queue. When `MediaSource` is reachable again, drain it (in order, deduped per item by latest position).
8. **Storage settings surface** — a new section in Settings: current usage bar, configurable cap (8 / 16 / 32 / 64 GB / Custom), "Clear all downloads" with confirmation, per-title size + Keep toggle.
9. **Eviction policy** — least-recently-watched first; "Keep" toggle pins a title so eviction skips it; eviction runs lazily, when the user starts a new download that would exceed the cap.

Each item is a single PR.

```
                   ├─► Download UI (Detail) ─┐
DownloadStore    ──┤                          ├─► Downloads tab
                   └─► DiskBudget ────────────┘
                                              │
                                              ▼
                                     Offline playback path
                                              │
                                              ▼
                                       Resume outbox
                                              │
                                              ▼
                                   Storage settings + eviction
```

`DownloadManager` + `DownloadStore` + `DiskBudget` ship as one PR (they're cohesive). The UI surfaces follow. The outbox + eviction land last — they're the bits that turn "downloads work" into "downloads feel right."

## Background `URLSession` plumbing

The whole point is **resumable, survives-app-kill** downloads. That means a `URLSession` with a `background` configuration, identified by a stable string (`cz.zmrhal.aether.downloads`).

- `URLSessionDownloadDelegate` callbacks fire even when the app was relaunched into the background. The delegate routes events to the `DownloadManager` actor via a continuation-bridged adapter (UIKit's URLSession delegate is not `Sendable`-clean; the adapter is the seam).
- Progress is throttled — emit at most ~4 events/sec per task so SwiftUI doesn't redraw every byte.
- Errors are retried with exponential backoff up to N attempts; after that the download flips to `.failed(reason)` and the UI offers retry.
- `application(_:handleEventsForBackgroundURLSession:completionHandler:)` is wired in `AetherApp` to keep the background session warm. iOS-specific.

## Sandboxed paths only

Per `AGENTS.md`, downloads land **inside the app's documents/caches directory**, never absolute paths.

```
<app sandbox>/Library/Caches/Downloads/<source>/<itemID>/<filename>
```

- The download file is the *only* place a stream URL touches the disk.
- `AppGroupContainer` isn't needed yet (no extension consumes downloads). When Top Shelf or a Watch extension lands, this moves to an app group.
- Encryption: none. These are the user's own files on the user's own device — encrypting them locally adds friction without a clear threat to defend against.

## Disk budget UI

Settings → Storage gets one section, three rows + one bar.

- **Bar** — total used vs cap, with per-title slices labeled by size (largest first; smaller ones collapse into "Other").
- **Cap row** — picker: 8 GB (default) / 16 / 32 / 64 / Custom.
- **Per-title rows** — title + size + Keep toggle + Remove. Selecting any opens the title's Detail screen with the Download action pre-focused on Remove.
- **Clear all** — destructive button at the bottom; confirms once, then evicts.

The bar uses `Tokens.Palette.accent` for the user's titles and a tertiary fill for system overhead. No spinner during clear — a skeleton + the bar shrinking is enough motion.

## Resume outbox

```
PlaybackSession writes resume ──┐
                                ▼
                  ResumeStore (local cache + outbox)
                                │
            ┌───────────────────┴───────────────────┐
            ▼                                       ▼
      cache lookup                       outbox queue (SwiftData)
   for Detail / Home                              │
                                                  ▼
                              On `MediaSource.becameReachable`:
                              drain in order, dedup per item by
                              `updatedAt` (latest wins).
```

- Resume writes are local-first regardless of network state.
- The outbox is a single SwiftData table keyed by `(source, itemID)`; only the latest position per key survives a dedup pass.
- "Reachable" is observed via the `MediaSource` — for Plex, a successful `/identity` probe; for Synology, a successful session ping.

## What this milestone is *not*

- **Selective downloads inside a show.** 0.3 downloads whole `MediaItem`s — a movie, an episode. Whole-show download (queue every episode) is a 0.4+ idea, not here.
- **Auto-download on cellular.** Off by default and not in 0.3 UI. A future Settings toggle adds it; until then, downloads only start when the network reachability check says Wi-Fi.
- **Cross-device download sync.** A download on iPhone doesn't appear on Apple TV. The architecture supports it later via the same outbox model, but it's not in 0.3.
- **Encrypted local storage.** Out — see *Sandboxed paths* above.

## Open questions

- **Subtitle / audio track persistence offline.** If the user picks a non-default subtitle / audio track and pauses, does that selection survive going offline? Plex serves transcoded HLS variants per language — the simplest answer is "download with default tracks; let the user re-pick offline if AVPlayer supports it on the local file." Verify which container/track combos AVPlayer can still switch in.
- **Transcoded downloads vs. direct file.** For incompatible containers (MKV), download the universal transcoder output (HLS) or the original MKV and rely on AVPlayer being able to play it post-download? HLS is friendlier for AVPlayer; MKV is smaller and lossless. Default: download HLS. Revisit if storage cost is too high.
- **Where the Downloads surface lives.** A dedicated tab vs. a section on Home vs. a Settings entry. tvOS suggests a tab; iOS could go either way. Decide before the UI lands; the right answer is probably a top-level tab on every platform once 0.3 ships.
- **Synology offline.** Synology direct-play URLs are scoped by session cookies — those cookies expire. The download needs to capture a long-lived signed URL or the user re-auths after a long offline gap. Spike during the Synology connector work in 0.2 to know which is true.

## Notes for future me

- The most important UX property of 0.3 is **honesty**. If a title isn't actually playable offline (codec, DRM, partial download), say so on the Detail screen *before* the user is on the plane. Never silently fall back to streaming when the user expected offline.
- Disk pressure on small phones is real. The default cap of 8 GB is conservative on purpose; people who travel a lot will lift it.
- This is the milestone where the in-memory `ResumeStore` becomes a SwiftData store. The seam already exists; don't reshape `ResumeStore`'s API when porting — just move the storage.
