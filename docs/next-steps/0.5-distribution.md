# 0.5 Distribution: Internal TestFlight

This is the working checklist for getting Aether into internal TestFlight using
Xcode Cloud.

Scope for the first pass:

- internal testers only
- Xcode Cloud archive and upload
- iOS first, then tvOS and visionOS after signing and platform verification
- no public links, no external beta review, no App Store submission yet

## Current repo readiness

Already in place:

- XcodeGen is the project source of truth in `project.yml`.
- `Aether` has a shared scheme with build, test, and archive actions.
- iOS unit tests pass locally on the simulator.
- App targets exist for iOS, tvOS, and visionOS.
- Bundle identifiers are defined:
  - iOS: `cz.zmrhal.aether`
  - tvOS: `cz.zmrhal.aether.tvos`
  - visionOS: `cz.zmrhal.aether.visionos`

Important gaps before Xcode Cloud can be trusted:

- `Aether.xcodeproj` is generated and ignored by git. Xcode Cloud needs a
  post-clone script that installs XcodeGen and runs `xcodegen generate`, or we
  must intentionally start committing the generated project. Keep the generated
  project out of git unless Xcode Cloud makes that impossible.
- `CFBundleVersion` is currently static (`"1"`). Repeated TestFlight uploads
  need a unique build number per marketing version. Prefer Xcode Cloud's build
  number for distributed builds.
- Signing configuration needs to be confirmed against the Apple Developer team
  in App Store Connect/Xcode Cloud. Do not rely on local generated
  `Aether.xcodeproj` signing settings unless they are also represented in
  `project.yml` or configured by Xcode Cloud managed signing.
- Local tvOS verification is blocked until the matching tvOS platform is
  installed in Xcode. Do not make tvOS the first required TestFlight product.

## Apple account setup

Required before the first workflow:

- Active Apple Developer Program membership.
- App Store Connect access with permission to create/manage the app record and
  internal TestFlight groups. Apple lists Account Holder, Admin, App Manager,
  Developer, or Marketing as eligible roles for adding internal testers.
- Git provider connected to Xcode Cloud for this repository.
- App record in App Store Connect for Aether.
- Registered bundle IDs in Certificates, Identifiers & Profiles for every
  platform we plan to archive:
  - `cz.zmrhal.aether`
  - `cz.zmrhal.aether.tvos`
  - `cz.zmrhal.aether.visionos`
- Capabilities reviewed per target. Today that mostly means normal app signing
  plus iOS background audio. Add associated domains, iCloud, push, or other
  entitlements only when the app really needs them.

## Internal testers

Create one internal group first:

- Name: `Aether Internal`
- Members: only App Store Connect users who should see early builds.
- Limit: internal testing is for up to 100 App Store Connect users with access
  to the app.
- Build lifetime: TestFlight builds are available for 90 days.

Operational note: Apple says builds created by Xcode Cloud must be manually
added to internal groups in App Store Connect. Keep automatic distribution off
for the first build so we can inspect the archive before inviting everyone.

Internal-only means:

- no external testers
- no public link
- no TestFlight beta app review path yet
- no customer/App Store release eligibility from the internal-only build

## Xcode Cloud workflow

Create the first workflow from Xcode after generating and opening the project:

- Product: `Aether`
- Workflow name: `Internal TestFlight`
- Start condition: manual only for the first run
- Branch: `main`
- Environment: latest stable Xcode that includes the required iOS 26 SDK
- Actions:
  - build `Aether`
  - test `Aether` on one current iPhone simulator
  - archive `Aether`
- Post-action:
  - distribute/archive to TestFlight for internal testing
- Notifications:
  - notify on failed builds and failed tests

After the first successful internal build:

- Add a branch-change start condition for `main`.
- Keep archive/distribution manual or gated until signing and build-number
  behavior is boring.
- Add tvOS as a second workflow or product after the platform and signing path
  are verified.
- Add visionOS after the app record and archive destination are confirmed.

## XcodeGen in Xcode Cloud

Because the generated `.xcodeproj` is intentionally not checked in, add a
post-clone script before the first cloud build:

```sh
#!/bin/sh
set -eu

brew install xcodegen
xcodegen generate
```

Expected path:

```text
ci_scripts/ci_post_clone.sh
```

Required follow-up:

```bash
chmod +x ci_scripts/ci_post_clone.sh
```

Apple requires Xcode Cloud custom scripts to live in `ci_scripts` and use the
recognized names (`ci_post_clone.sh`, `ci_pre_xcodebuild.sh`, or
`ci_post_xcodebuild.sh`). The post-clone timing is the right place for
XcodeGen because Xcode Cloud clones the repository before it runs `xcodebuild`.

## Version and build numbers

Current project values:

- Marketing version: `0.1.0`
- Build number: `1`

Before the first internal upload:

- Decide whether the first internal train is still `0.1.0` or should become
  `0.2.0`.
- Keep marketing version human-managed in `project.yml`.
- Let Xcode Cloud provide the distributed build number, or wire
  `CFBundleVersion` to a build setting that Xcode Cloud can override.
- Do not upload two builds with the same
  `CFBundleShortVersionString` + `CFBundleVersion` pair.

Apple notes that Xcode Cloud increments integer build numbers and App Store
Connect uses the Xcode Cloud build number when a cloud build is distributed
with TestFlight or released on the App Store.

## First internal release checklist

Before running the workflow:

- Generate the project locally with `xcodegen generate`.
- Open `Aether.xcodeproj` and confirm the `Aether` scheme is visible.
- Confirm signing team for the iOS target.
- Run the iOS test suite locally.
- Add the Xcode Cloud post-clone script for XcodeGen.
- Commit `project.yml`, docs, and `ci_scripts`.

After the workflow archives:

- Wait for App Store Connect processing to finish.
- Add the processed build to `Aether Internal`.
- Add concise "What to Test" notes:
  - Plex PIN sign-in
  - server discovery on home Wi-Fi
  - movie playback with direct play
  - MKV playback through Plex transcoding
  - show -> season -> episode browsing
  - player dismissal and auto-hiding chrome
  - Settings -> Sign Out of Plex
- Install from TestFlight on a real iPhone and iPad.
- Record any crash, launch, signing, or networking failures before inviting
  the wider internal group.

## Smoke test matrix

Minimum device pass for internal testing:

| Surface | Device | What to verify |
|---------|--------|----------------|
| iPhone | current personal device | Sign-in, browsing, playback, sign-out |
| iPad | real device or simulator first | Layout, rotation, playback controls |
| Apple TV | after tvOS signing/platform setup | Focus, remote navigation, playback |
| Vision Pro | later | Windowed launch and shared UI sanity |

## Known blockers and decisions

- Xcode Cloud cannot build this repo until generated project creation is part
  of the cloud workflow.
- tvOS local build verification currently depends on installing the matching
  tvOS platform in Xcode.
- The first internal TestFlight should be iOS-only unless there is a specific
  need to test Apple TV immediately.
- External TestFlight, beta review, screenshots, privacy nutrition labels,
  app privacy, accessibility nutrition labels, support URL, and App Store copy
  belong to the broader 0.5 release track, not the first internal build.
