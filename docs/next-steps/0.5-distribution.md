Scope for the first pass:

- internal testers only
- Xcode Cloud archive and upload

## Current repo readiness

Already in place:

- **Single multiplatform target.** `project.yml` declares one `Aether` application target with `supportedDestinations: [iOS, tvOS, visionOS]`. iPad is implicit under iOS. Xcode picks the right Info.plist values and asset catalog variant per destination at archive time.
- **Bundle identifier:** `cz.zmrhal.aether` ‚Äî one App ID covers every destination.
- **Marketing version + build number live in build settings** (`MARKETING_VERSION = "0.1.0"`, `CURRENT_PROJECT_VERSION = "1"`) so Xcode's General ‚Üí Identity panel reads them. Info.plist references them via `$(MARKETING_VERSION)` / `$(CURRENT_PROJECT_VERSION)` substitution.
- **`ci_scripts/ci_post_clone.sh`** ‚Äî Xcode Cloud installs XcodeGen via Homebrew and runs `xcodegen generate` in the cloned workspace. The `.xcodeproj` stays out of git.
- **`ci_scripts/ci_pre_xcodebuild.sh`** ‚Äî patches `CFBundleVersion` in the single Info.plist to `$CI_BUILD_NUMBER` (PlistBuddy) so every cloud archive ships a unique build number without manual bumps.
- **`ITSAppUsesNonExemptEncryption: false`** in Info.plist so App Store Connect skips the encryption export compliance question every build.
- **Placeholder layered app icons** for every destination, sharing the canonical name `AppIcon` so a single `ASSETCATALOG_COMPILER_APPICON_NAME` setting resolves the right type per destination:
  - iOS / iPadOS ‚Üí `Assets.xcassets/AppIcon.appiconset` (single 1024√ó1024 PNG).
  - tvOS ‚Üí `Assets.xcassets/AppIcon.brandassets` (App Icon - App Store 1280√ó768, App Icon - Home Screen 400√ó240, Top Shelf 1920√ó720; each layered icon is 3 imagestacklayer Back/Middle/Front stacks).
  - visionOS ‚Üí `Assets.xcassets/AppIcon.solidimagestack` (Back / Middle / Front layers at 1024√ó1024).
  Same indigo gradient + accent play triangle across all three. Regenerable any time via `swift Tools/generate-app-icon.swift`. Designed icons replace them before external review.
- iOS unit tests pass locally on the simulator (Xcode 26.4 / iPhone 17).

Remaining gaps:

- **Apple Developer account, app record, internal tester group, Xcode Cloud workflow** ‚Äî all live in Apple's consoles, not in this repo. See *Apple-side checklist* below.
- **Signing** ‚Äî Xcode Cloud's managed signing is the recommended path for the first internal upload. `project.yml` pins the Apple Developer Team ID (`8PW5FWH7P2`) and automatic signing so the generated Xcode project carries the same signing identity locally and in Xcode Cloud.
- **Local tvOS / visionOS archive verification** depends on installing the matching simulator runtimes in Xcode. Not a cloud blocker ‚Äî Xcode Cloud has all three SDKs.

## Apple-side checklist (one-time)

These all have to be done by a human in Apple's web consoles ‚Äî the repo can't do them on its behalf. Do them once, in this order; later builds reuse all of it.

1. **Apple Developer Program** ‚Äî confirm membership is active.
2. **App Store Connect access** ‚Äî log in as Account Holder / Admin / App Manager so you can create the app record and TestFlight group.
3. **Register the bundle ID** in *Certificates, Identifiers & Profiles* ‚Üí *App IDs*:
   - **App ID:** `cz.zmrhal.aether`
   - **Description:** *Aether*
   - **Capabilities:** Background Modes (Audio, AirPlay, and Picture in Picture). Everything else off for now.
   - That single App ID covers iPhone + iPad + Apple TV + Apple Vision Pro because the underlying target is multiplatform.
4. **Create the app record** in App Store Connect (*My Apps ‚Üí +*):
   - **Platforms:** *iOS, tvOS, visionOS* (selectable as a single multi-destination app).
   - **Bundle ID:** `cz.zmrhal.aether`.
   - **Name:** *Aether*.
   - **SKU:** `aether` (or whatever you prefer).
   - This is **one** app record; the destinations live underneath it.
5. **Connect the GitHub repository to Xcode Cloud** from Xcode ‚Üí *Integrate ‚Üí Manage Workflows* ‚Üí connect *Wenzlik/Aether*.
6. **Create the internal tester group**: TestFlight ‚Üí *Internal Testing ‚Üí +* ‚Üí name `Aether Internal`. Add the App Store Connect users who should see early builds (limit: 100; build lifetime: 90 days).
7. **Create one Xcode Cloud workflow.** Because the target is multiplatform, one workflow can archive every destination:
   - **Workflow name:** `Internal TestFlight`.
   - **Branch:** `main`.
   - **Environment:** latest stable Xcode that includes iOS 26 / tvOS 26 / visionOS 26 SDKs.
   - **Actions:**
     - *Build* ‚Üí product *Aether*, configuration *Debug*, any iPhone simulator (warms caches; safe to drop if you trust the post-build).
     - *Test* ‚Üí run `AetherTests` on a current iPhone simulator.
     - *Archive (iOS)* ‚Üí product *Aether*, destination *Any iOS Device*, configuration *Release*.
     - *Archive (tvOS)* ‚Üí product *Aether*, destination *Any Apple TV Device*, configuration *Release*.
     - *Archive (visionOS)* ‚Üí product *Aether*, destination *Any Apple Vision Pro Device*, configuration *Release*.
   - **Post-action:** distribute each archive to TestFlight (internal).
   - **Start condition:** *Manual* for the first run; flip to *Branch Change ‚Üí main* once the first build is happy.
   - **Signing:** leave `Automatically manage signing` on (Xcode Cloud managed signing).
   - **Notifications:** notify on failed builds + failed tests.
8. **Keep automatic distribution off for the first archive of each destination.** Apple's docs are explicit: builds from Xcode Cloud need to be manually added to internal groups for the first round so the archive can be inspected. Once stable, flip distribution to automatic.

## Version and build numbers

| What | Where | Notes |
|---|---|---|
| Marketing version | `MARKETING_VERSION` in `project.yml` `settings.base` (today: `"0.1.0"`) | Human-bumped between trains. Surfaces as `CFBundleShortVersionString` via `$(MARKETING_VERSION)` in Info.plist. |
| Build number | `CURRENT_PROJECT_VERSION` in `project.yml` (today: `"1"`) | Default for local builds. Xcode Cloud overwrites at archive time via `ci_pre_xcodebuild.sh` using `$CI_BUILD_NUMBER`. Surfaces as `CFBundleVersion` via `$(CURRENT_PROJECT_VERSION)` in Info.plist. |

Never upload two builds with the same `CFBundleShortVersionString` + `CFBundleVersion` pair ‚Äî App Store Connect rejects duplicates.

Before the first internal upload, decide whether the first train stays at `0.1.0` or jumps to `0.2.0`. Either is fine for internal testing; the call only matters for what testers see in TestFlight.
