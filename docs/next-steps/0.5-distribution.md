# 0.5 Distribution: Internal TestFlight

This is the working checklist for getting Aether into internal TestFlight using
Xcode Cloud.

Scope for the first pass:

- internal testers only
- Xcode Cloud archive and upload
- **iOS, tvOS, and visionOS** all in the first wave (placeholder layered icons are now generated for tvOS and visionOS, so all three can archive — see *Repo readiness* below)
- no public links, no external beta review, no App Store submission yet

## Current repo readiness

Already in place:

- XcodeGen is the project source of truth in `project.yml`.
- `Aether`, `Aether-tvOS`, and `Aether-visionOS` schemes all have build / archive actions; the iOS scheme additionally runs `AetherTests`.
- iOS unit tests pass locally on the simulator (Xcode 26.4 / iPhone 17).
- App targets exist for iOS, tvOS, and visionOS.
- Bundle identifiers are defined:
  - iOS: `cz.zmrhal.aether`
  - tvOS: `cz.zmrhal.aether.tvos`
  - visionOS: `cz.zmrhal.aether.visionos`
- **`ci_scripts/ci_post_clone.sh`** — installs XcodeGen via Homebrew and runs `xcodegen generate` inside the cloud workspace.
- **`ci_scripts/ci_pre_xcodebuild.sh`** — patches `CFBundleVersion` on every platform's `Info.plist` to `$CI_BUILD_NUMBER` so repeated TestFlight uploads get unique build numbers without manual bumps.
- **`ITSAppUsesNonExemptEncryption: false`** in every target's Info.plist so App Store Connect skips the encryption export compliance question for each build (Aether uses only standard Apple URLSession HTTPS).
- **Placeholder layered app icons** for tvOS (Brand Asset — App Icon - App Store at 1280×768, App Icon - Home Screen at 400×240, Top Shelf Image at 1920×720) and visionOS (Solid Image Set — Back / Middle / Front layers at 1024×1024). Generated by `swift Tools/generate-app-icon.swift`, regenerable any time. Designed icons replace them before external review; for internal TestFlight they unblock all three archives today.

Remaining gaps:

- **Apple Developer account, app records, bundle IDs registered, internal tester group, Xcode Cloud workflows** — all live in Apple's consoles, not in this repo. See *Apple-side checklist* below.
- **Signing** — Xcode Cloud's managed signing (Apple Distribution managed by cloud) is the recommended path for the first internal upload; no `DEVELOPMENT_TEAM` is hardcoded in `project.yml` on purpose so the cloud picks the team from the App Store Connect record.
- **Local tvOS / visionOS verification** depends on installing the matching simulator runtimes in Xcode. Not a blocker for Xcode Cloud, which has them all.

## Apple-side checklist (one-time)

These all have to be done by a human in Apple's web consoles — the repo can't do them on its behalf. Do them once, in this order; later builds reuse all of it.

1. **Apple Developer Program** — confirm the membership is active. The current team is what every bundle ID below has to belong to.
2. **App Store Connect access** — log in as Account Holder / Admin / App Manager so you can create app records and TestFlight groups.
3. **Register bundle IDs** in *Certificates, Identifiers & Profiles*, each as an *App ID*:
   - `cz.zmrhal.aether` (iOS / iPadOS — Game Controllers, Maps, Push: all off; Background Modes: Audio, AirPlay, and Picture in Picture **on**)
   - `cz.zmrhal.aether.tvos` (tvOS — no special capabilities yet)
   - `cz.zmrhal.aether.visionos` (visionOS — no special capabilities yet)
4. **Create the app records** in App Store Connect (*My Apps → +*). One record per bundle ID, three records total. Use the same SKU prefix (e.g. `aether-ios`, `aether-tvos`, `aether-visionos`) so they line up visually.
5. **Connect the GitHub repository to Xcode Cloud** from Xcode → *Integrate → Manage Workflows* → connect *Wenzlik/Aether*.
6. **Create the internal tester group**: TestFlight → *Internal Testing → +* → name `Aether Internal`. Add only people who should see early builds. Limit is 100 App Store Connect users; build lifetime is 90 days.
7. **Create three Xcode Cloud workflows**, one per product:
   - **Aether (iOS)** — branch `main`, environment latest stable Xcode with iOS 26 SDK, actions: build + test (one current iPhone simulator) + archive, post-action: TestFlight (internal). Start condition: **Manual** for the first run; switch to *Branch Change → main* once the first build is happy.
   - **Aether-tvOS** — branch `main`, same Xcode environment, actions: build + archive (no test action — tests are iOS-only in the current scheme), post-action: TestFlight (internal). Manual for the first run.
   - **Aether-visionOS** — branch `main`, same Xcode environment, actions: build + archive, post-action: TestFlight (internal). Manual for the first run.

   Each workflow uses Xcode Cloud's managed signing (default) — leave `Automatically manage signing` on.
8. **Keep automatic distribution off for the first archive of each platform.** Apple's docs are explicit: builds from Xcode Cloud need to be manually added to internal groups for the first round so the archive can be inspected. Once it's stable, flip distribution to automatic.

## XcodeGen in Xcode Cloud

Because the generated `.xcodeproj` is intentionally not checked in, add a
post-clone script before the first cloud build:

```sh
#!/bin/sh
set -eu

brew install xcodegen
xcodegen generate
```

Expected path:

```text
ci_scripts/ci_post_clone.sh
```

Required follow-up:

```bash
chmod +x ci_scripts/ci_post_clone.sh
```

Apple requires Xcode Cloud custom scripts to live in `ci_scripts` and use the
recognized names (`ci_post_clone.sh`, `ci_pre_xcodebuild.sh`, or
`ci_post_xcodebuild.sh`). The post-clone timing is the right place for
XcodeGen because Xcode Cloud clones the repository before it runs `xcodebuild`.

- Let Xcode Cloud provide the distributed build number, or wire
  `CFBundleVersion` to a build setting that Xcode Cloud can override.
- Do not upload two builds with the same
  `CFBundleShortVersionString` + `CFBundleVersion` pair.

Apple notes that Xcode Cloud increments integer build numbers and App Store
Connect uses the Xcode Cloud build number when a cloud build is distributed
with TestFlight or released on the App Store.

## First internal release checklist

Before running the workflow:

- Generate the project locally with `xcodegen generate`.
- Open `Aether.xcodeproj` and confirm the `Aether` scheme is visible.
- Confirm signing team for the iOS target.
- Run the iOS test suite locally.
- Add the Xcode Cloud post-clone script for XcodeGen.
- Commit `project.yml`, docs, and `ci_scripts`.

After the workflow archives:

- Wait for App Store Connect processing to finish.
- Add the processed build to `Aether Internal`.
- Add concise "What to Test" notes:
  - Plex PIN sign-in
  - server discovery on home Wi-Fi
  - movie playback with direct play
  - MKV playback through Plex transcoding
  - show -> season -> episode browsing
  - player dismissal and auto-hiding chrome
  - Settings -> Sign Out of Plex
- Install from TestFlight on a real iPhone and iPad.
- Record any crash, launch, signing, or networking failures before inviting
  the wider internal group.
