# Aether — Unified Library + Navigation Refactor (0.5.0)

> **Status:** architecture plan / doc-only PR (pre-implementation). Cross-platform
> (iOS / iPadOS / tvOS / visionOS).
> **Goal:** the *source* becomes an implementation detail. Users think
> **Movies / TV Shows / Downloads**, not Plex / Jellyfin / offline. One title,
> one row, multiple sources behind it.

This is the next major milestone after Cinema V1 and the foundation for 0.5.0.
It touches Home, Library, Search, Detail, Downloads, Settings, navigation, and
adds an Emby connector — so it lands as a **sequence of PRs**, not one.

---

## 1. Current state (what we build on)

Aether is already partly source-agnostic, which makes this evolution additive
rather than a rewrite:

- **`MediaItem`** (`AetherCore/Models`) is a unified, source-agnostic value type;
  views/navigation/playback never branch on the source.
- **`MediaSource`** is the shared connector protocol (Plex, Jellyfin today;
  Synology stub). Each produces `MediaItem`s and answers `resolvePlayback` /
  `downloadURL`.
- **`MediaSourceID`** enum scopes every id: `.plex(serverID:)`,
  `.jellyfin(serverID:)`, `.synology`, `.mock`.
- **`AppSession`** today holds **one active source** (`activeSourceKind`) and
  points `source: (any MediaSource)?` at it. Switching happens in Settings.
- Downloads: `DownloadStore` / `DownloadObserver` track completed downloads keyed
  by `MediaID`. Offline playback already overrides in `PlaybackSession`.

**The gap:** there is exactly **one active source at a time** and **no merged
feed**. ARCHITECTURE.md explicitly flagged this as layerable later — that "later"
is now.

**The two hard problems** Unified Library introduces:
1. **Aggregation** — query *all* connected sources and merge their results.
2. **Identity** — recognise that "Blade Runner 2049" on Plex, on Jellyfin, and
   offline are the *same title* so it appears once. This needs **external IDs**
   (TMDB/IMDB/TVDB), which we don't currently parse.

## 2. Unified media model

Layer **on top of** `MediaItem` — don't replace it. A `UnifiedMediaItem`
aggregates one-or-more per-source `MediaItem`s for the same title.

```swift
struct UnifiedMediaItem: Identifiable, Hashable, Sendable {
    let id: UnifiedID            // stable, derived from external IDs (see §4)
    let title: String
    let originalTitle: String?
    let year: Int?
    let overview: String?
    let posterURL: URL?          // chosen from the preferred source
    let backdropURL: URL?
    let type: MediaType          // .movie / .show  (episodes resolve within a show)
    let sources: [UnifiedSource] // every source that has this title
    let playbackProgress: Double?// unified resume (max across sources)
    var isDownloaded: Bool       // any source has a completed download
}

struct UnifiedSource: Identifiable, Hashable, Sendable {
    let id: MediaSourceID        // reuse existing scoping
    let kind: MediaSourceKind    // .offline / .plex / .jellyfin / .emby
    let item: MediaItem          // the concrete per-source item (playback/download)
    let serverName: String?
    let quality: String?         // e.g. "4K HDR" from MediaInfo, for display
    var playable: Bool
}

enum MediaSourceKind: Int, Comparable, Sendable {
    case offline = 0, plex = 1, jellyfin = 2, emby = 3   // rawValue = priority
}
```

Notes:
- `UnifiedSource.item` keeps the real `MediaItem`, so playback/download reuse the
  existing `MediaSource.resolvePlayback` / `downloadURL` unchanged.
- `MediaSourceKind` raw value **is** the priority order (offline first).
- The spec's `UnifiedMediaItem`/`MediaSource` names are honoured; we just align
  them with Aether's existing `MediaItem` / `MediaSourceID` instead of duplicating.

## 3. Aggregation layer

A new `UnifiedLibrary` actor in `AetherCore/Library/` is the single entry point
the UI talks to. It owns the set of **connected** sources (not one active one).

```
Views ──► UnifiedLibrary (actor)
              ├─ fan-out: query every connected MediaSource concurrently
              ├─ merge + dedupe (§4) into [UnifiedMediaItem]
              ├─ fold in offline (DownloadStore) as a source
              └─ stale-while-revalidate cache per (library, sort, page)
```

- **Fan-out, fault-tolerant:** one source failing/slow must not blank the feed —
  return what resolved, mark the rest pending (mirrors how a finder fans out).
- **AppSession change:** replace the single `activeSourceKind`/`source` with a
  list of connected sources + the `UnifiedLibrary`. The single-source code paths
  (Detail, player, downloads) keep working because they operate on a concrete
  `MediaItem` from a `UnifiedSource`.
- **Caching is essential** — every aggregation is N network calls. Reuse the
  stale-while-revalidate approach from ARCHITECTURE.md's caching section.

## 4. Deduplication

Two per-source items are the **same title** when their external IDs match.

**ID sources (must be parsed — new work):**
- **Plex:** the `Guid` tags on metadata (`com.plexapp.agents.imdb`, and the newer
  `tmdb://` / `imdb://` / `tvdb://` `Guid` array). Surface into `MediaItem`.
- **Jellyfin:** `ProviderIds` dict (`Tmdb` / `Imdb` / `Tvdb`).
- **Offline:** inherits the IDs of the item it was downloaded from.

Add to `MediaItem`:

```swift
struct MediaGuids: Hashable, Sendable {
    var tmdb: String?; var imdb: String?; var tvdb: String?
}
let guids: MediaGuids   // new field, populated by connectors
```

**Match precedence:** TMDB → IMDB → TVDB → **title + year** fallback (normalised
title, ±0 year) only when no shared external ID exists. The fallback is
deliberately conservative to avoid false merges.

`UnifiedID` is derived from the strongest available external ID (so it's stable
run-to-run); title+year-only items get a synthesised id and never merge across
sources.

> This is the linchpin. Without external IDs the whole feature degrades to fuzzy
> title matching. Parsing GUIDs is **Phase 1** and gated behind tests.

## 5. Source priority + playback resolution

When the user plays a `UnifiedMediaItem`, pick the best `UnifiedSource`:

```
preferred = sources.filter(\.playable).min(by: kind)   // offline < plex < jellyfin < emby
```

- **Automatic:** offline → Plex → Jellyfin → Emby. Offline first never burns
  bandwidth and works on a plane.
- **Fallback:** if the preferred source fails to resolve, fall to the next.
- **Manual override:** the Detail "Available Sources" section lets the user pin a
  source for this playback; that choice routes through the same
  `MediaSource.resolvePlayback` on the chosen `UnifiedSource.item`.

Offline as a source: a completed `DownloadStore` job for any of the title's
`MediaID`s yields a synthetic `.offline` `UnifiedSource` whose `item.streamURL`
is the local file (the offline path `PlaybackSession` already handles).

## 6. UI changes

### Detail
- **Keep** the compact rows: Audio / Subtitles / Quality.
- **Add** a `Source` disclosure row showing the active source (`Plex` / `Offline`).
- **Add** an **Available Sources** section listing every source behind the title
  (Offline / Plex / Jellyfin / Emby) with server name + quality; tap to override.
- **Available Offline** badge when a completed download exists — downloads are a
  source, not a separate ecosystem.

### Home (unified rails)
- `Continue Watching`, `Recently Added`, `Movies`, `TV Shows`, `Downloaded` — all
  fed by `UnifiedLibrary`. No Plex- or Jellyfin-specific rails.
- `HomeFeedBuilder` is refactored to consume unified items.

### Search (unified)
- Query all connected sources concurrently, dedupe, return `UnifiedMediaItem`s —
  one movie, multiple sources behind it.

### Discover
- Promote to all platforms (iOS / iPadOS / visionOS / tvOS). Home for trending /
  recommendations / new releases and (later) catalog-only connectors.

| Platform | Tabs |
|---|---|
| iPhone / iPad | Home · Library · Discover · Search · Settings |
| tvOS | Home · Library · Discover · Search · Settings |
| visionOS | Home · Library · Discover · Search · **Storage** · Settings |

  **Settings → Downloads**. Kept as a tab on **visionOS** (offline has more value
  there).
- **Discover** becomes a first-class tab everywhere (it exists on tvOS today).
- `RootTabView` already branches tabs per platform (`#if os(tvOS)` Discover vs
  Storage) — this generalises that.

## 9. Emby connector

Add `EmbyMediaSource` as a first-class `MediaSource`. Emby's API is close to
Jellyfin's (shared lineage), so it largely mirrors `JellyfinMediaSource`
(auth, items, `resolvePlayback`, `downloadURL`, GUID parsing via `ProviderIds`).
**No source gets special UI treatment** — it's just another `UnifiedSource`.

**Future catalog-only connectors** (Netflix / Disney+ / Prime / Apple TV
*catalogs*, not playback) must slot in: model a source as
`playable: false` + catalog metadata, so Discover/Search can surface a title even
when Aether can't play it. The `UnifiedSource.playable` flag already expresses
this.

## 10. Migration notes

- **Existing users:** `AppSession` migrates `activeSourceKind` → "all connected
  sources." No re-sign-in: existing Plex/Jellyfin tokens/servers are reused; both
  become connected simultaneously.
- **Resume points** stay keyed by `MediaID` (per-source); unified progress is the
  max across a title's sources — no data migration.
- **Downloads** unchanged on disk; they're just surfaced as a source.
- First unified launch may show brief "merging" as caches warm; sources appear
  incrementally (fault-tolerant fan-out).

## Risks / decisions to confirm

- **External IDs availability.** If a server's metadata lacks TMDB/IMDB/TVDB IDs
  (un-matched library items), those titles fall to title+year and may not merge.
  Acceptable, but worth confirming the tolerance.
- **Aggregation cost / pagination.** Merging paginated lists across sources with
  different orders is non-trivial; Phase 1 should define the paging contract.
- **One-active-source removal** touches `AppSession` broadly — needs care so
  Detail/player/downloads keep working (they operate on concrete `MediaItem`s, so
  the blast radius is the session + feed builders, not playback).
- **Numbering:** confirm this is 0.5.0 and how the existing "0.4 Premium UX" /
  "0.5 Distribution" roadmap entries reconcile.
